Motion Data Processing Explained

Motionprint Ergo processes raw inertial sensor data from Xsens motion capture systems to generate actionable ergonomic insights. This article explains what happens between importing an MVNX file and seeing a risk score — how sensor data is structured, how it is converted into joint angles, how postures are classified, and how the metrics fed into NIOSH and KIM-LHC assessments are derived.

1 Raw Data Input

The input to every Motionprint Ergo assessment is an MVNX file — an XML-format file exported by Xsens MVN Analyze or Animate at the end of a motion capture session. This file contains everything the Xsens MVN Engine computed from the raw IMU sensor signals during the recording.

What the sensors capture

Each Xsens sensor unit contains an accelerometer and a gyroscope. The MVN Engine fuses the raw signals from all sensors worn on the body to compute a consistent, drift-corrected representation of the full body pose at every frame. The data written to the MVNX file is the output of this fusion process — not the raw IMU signals themselves.

Data channels in the MVNX file

Motionprint Ergo reads the following data channels from each MVNX file:

  • 3D segment positions — world-space coordinates (x, y, z) for the centre of mass of each body segment, at every frame
  • Segment orientations — quaternion (qw, qx, qy, qz) representing the rotation of each segment in 3D space
  • Joint angles — three-axis angles (flexion/extension, axial rotation, lateral bending) for each of the 22 joints
  • Centre of mass — the calculated whole-body centre of mass trajectory over time
  • Foot contacts — left and right foot contact flags indicating ground contact at each frame

Body model

Motionprint Ergo uses the standard Xsens 23-segment body model. Each segment has its own position and orientation tracked independently at every frame.

Supported formats

Motionprint Ergo primarily supports the MVNX format. For specific assessments, CSV input may also be supported. Preprocessing scripts are available for trimming recordings and regenerating derived variables when raw exports require adjustment before import.

2 Data Conversion

Before ergonomic metrics can be computed, the raw orientation and position data from the MVNX file must be converted into the angle and distance representations that assessment methods work with.

Quaternion to joint angles

Segment orientations in the MVNX file are stored as quaternions — a compact, gimbal-lock-free representation of 3D rotation. To compute joint angles, Motionprint Ergo converts relative quaternions between adjacent body segments into Euler angle representations using specific axis permutations that match the anatomical joint convention for each joint.

This produces the three-axis joint angles (flexion/extension, axial rotation, lateral bending or abduction/adduction) used in posture classification and biomechanical calculations. The Xsens MVN Engine applies filtering during the capture session itself to produce clean orientation data; Motionprint Ergo performs additional post-processing to ensure data quality and consistency across the full recording before passing it to the assessment pipeline.

Frame-by-frame processing

Every derived metric — posture angles, hand distances, joint loads, asymmetry values, and rolling risk scores — is computed independently for every frame in the recording. A 60-second recording at 60 Hz produces 3,600 data points per metric. This frame-level computation creates the full time-series dataset rendered as interactive graphs in the Analysis tab, allowing you to identify the exact moment in the recording where a high-risk posture occurred.

3 Posture Classification

Alongside angle computation, Motionprint Ergo classifies the worker's body posture for every frame. This classification is used in the KIM-LHC assessment — which scores posture across 10 body regions — and populates the posture breakdown view in the results screen.

Classification logic

Postures such as upright standing, trunk bending, lateral inclination, and trunk rotation are identified frame-by-frame using proprietary classification algorithms applied to the computed joint angles. Each body region is assigned to a posture category based on the angle ranges it falls within at that frame.

Posture type Classification threshold Body region
Trunk rotation Axial rotation > 10° L5 / T12 joint
Lateral trunk inclination Lateral bending > 10° L5 / T12 joint
Trunk flexion Flexion/extension angle ranges L5 / T12 joint

Transitions and boundary values

The classification engine is designed to handle transitions smoothly — posture categories are not assigned per-frame in isolation but take the movement trajectory into account to avoid spurious rapid switching between categories at boundary values. This ensures that brief postural fluctuations at a threshold boundary do not inflate the classified time in a higher-risk posture category.

4 Assessment Inputs

Once joint angles and posture classifications are established, Motionprint Ergo extracts the specific metrics required by the selected assessment method. These are passed to the NIOSH or KIM-LHC scoring modules to produce the final risk score.

Metrics extracted

  • Joint loads — estimated mechanical load at key joints (spine, shoulders, hips) based on segment positions and movement dynamics
  • Asymmetry angles — the angle of trunk asymmetry relative to the hands, computed at every frame of the lifting moment
  • Ergonomic risk scores — the intermediate sub-scores for each assessment factor, aggregated into the final risk score

Key derived variables

Several biomechanical variables required by the NIOSH Lifting Equation are derived automatically from the motion data, eliminating the need for manual measurement during or after the recording.

  • H — Horizontal distance: horizontal distance between the worker's ankle mid-point and the mid-point of the hands at lift origin, from 3D segment positions
  • V — Vertical hand height: height of the hands above the floor at lift origin, from the Y-axis position of the hand segments relative to the calibrated floor plane
  • D — Vertical travel distance: absolute vertical distance the hands travel between lift origin and destination
  • A — Asymmetry angle: angular deviation of the pelvis mid-sagittal plane from the hands mid-sagittal plane

Asymmetry multiplier (AM)

The asymmetry angle A is used to compute the Asymmetry Multiplier (AM), one of the six multipliers in the NIOSH Lifting Equation. A higher asymmetry angle results in a lower AM, which increases the overall Lifting Index.

NIOSH Asymmetry Multiplier AM = 1 − (0.0032 × A)

Where A is the asymmetry angle in degrees. An asymmetry angle of 0° gives AM = 1.0 (no penalty); at A = 135° the AM reaches its minimum value of 0.57. Values of A beyond 135° are capped at 135° per the NIOSH standard.

Note

H, V, D, and A are each computed for every frame of the lifting moment. The NIOSH calculation uses the values at the lift origin (start of the moment) for H, V, and A, and the full vertical travel range for D. Peak and mean values for each variable are available in the detailed output.

5 Error Handling

Motionprint Ergo validates the MVNX file and the motion data at multiple points in the processing pipeline. When issues are detected, the platform provides clear feedback rather than silently producing unreliable results.

File validation

Before any processing begins, the imported file is checked for:

  • Presence of all 23 required body segments
  • Valid joint angle data for the joints used in the selected assessment method
  • Consistent frame rate throughout the recording
  • Correct coordinate mapping and segment orientation format
  • Minimum frame count (recordings too short cannot produce a meaningful assessment)
  • No corrupted or truncated frame data

If validation fails, Motionprint Ergo displays a specific error message indicating which check failed and what needs to be corrected before re-importing.

Missing and noisy data

For cases where data is partially missing or contains sensor artefacts, the platform uses fallback mechanisms to handle edge cases gracefully. Where a reliable result cannot be produced, the platform defaults to a degraded output mode — for example, flagging affected metrics in the report rather than silently computing from incomplete data. User alerts are shown in the results view whenever a fallback has been applied, so you always know the confidence level of each output value.

Best practice

To minimise validation errors, always export MVNX files with the default Xsens MVN settings and ensure all 17 sensors are connected and calibrated before recording. Incomplete sensor sets will result in missing segment data.
